# Поведенческая модель аномалий (Risk-Based Step-Up)

Этот документ описывает, как в SCC работает модель оценки аномального поведения пользователя, какие математические методы применяются, как принимается решение о step-up, и как устроены механизмы защиты от ложных срабатываний.

## 1. Задача модели

Классическая проверка прав (RBAC/ACL) не отвечает на вопрос: «действует ли сейчас легитимный пользователь в привычном профиле поведения?».  
Модель добавляет zero-trust слой и рассчитывает `risk_score ∈ [0..1]` по потоку действий пользователя.

При превышении порога запускается `step-up`:

1. Проверка пароля.
2. Второй фактор: `TOTP` или `passkey`.

Если step-up повторно проваливается, включается эскалация временной блокировки:

1. 1 минута
2. 15 минут
3. 1 час

## 2. Поток данных и события

Модель работает на сервере, не доверяет UI, и опирается на фактические API-запросы.

Каждому запросу присваивается класс события:

- `view.docs` — чтение документов.
- `view.assets` — чтение активов/чувствительных реестров.
- `export.docs` — экспорт/действия экспорта.
- `mutation.high` — значимые изменения (POST/PUT/DELETE в критичных модулях).
- `denied.security` — отказ доступа (`401/403`) в защищенных маршрутах.
- `request.any` — прочие запросы (фон для интенсивности).

Данные пишутся в:

- `user_behavior_events`
- `user_behavior_risk_state`

## 3. Признаки (feature engineering)

Используются скользящие окна и исторические базовые уровни.

Короткие окна:

- `SensitiveViews5m`
- `Exports30m`
- `Denied10m`
- `Mutations5m`
- `Requests1m`

Исторические признаки (7 дней):

- `HistorySensitive`
- `HistoryExports`
- `HistoryDenied`
- `HistoryMutations`
- `HistoryRequests`
- `HistoryEvents`

Контекстные признаки:

- `IPNovelty ∈ {0,1}` — новый IP за период наблюдения.
- `RecentlyVerified` — недавно успешно пройденный step-up.

## 4. Математический метод

### 4.1 Нормализация через z-score

Для каждого счетчика в коротком окне вычисляется положительная z-оценка:

`z = max(0, (x - μ) / σ)`

Где:

- `x` — текущее значение в окне.
- `μ` — историческое среднее.
- `σ = sqrt(μ + 0.25)`, с нижней границей `σ >= 0.5` для численной устойчивости.

### 4.2 Линейный риск-скор (logit)

`raw = -3.4 + 0.85*z_sensitive + 1.2*z_exports + 1.4*z_denied + 0.7*z_mutations + 0.55*z_requests + 0.9*ip_novelty + 0.6*burst`

`burst = max(0, (SensitiveViews5m + Mutations5m - 12) / 6)`

Коэффициенты подобраны так, чтобы сильнее наказывать:

- отказ доступа,
- экспортную активность,
- всплески интенсивности,

и мягче учитывать обычный рост чтения/запросов.

### 4.3 Преобразование в вероятность

Используется сигмоида:

`score = 1 / (1 + e^(-raw))`

Итоговый `score` ограничивается: `clamp(score, 0, 1)`.

## 5. Anti-false-positive механизмы

После базового score применяются демпферы:

- Низкая активность (`totalRecent < 6`) -> `score *= 0.55`.
- Нет отказов/экспорта и IP не новый -> `score *= 0.75`.
- Недавний успешный step-up -> `score *= 0.65`.
- Cold-start ограничение: при малой истории и невысокой агрессии применяется мягкий ceiling `score <= 0.90`.

Это снижает риск ложных тревог при редкой/обычной работе.

## 6. Политика срабатывания step-up

Step-up запрашивается только при выполнении обоих условий:

1. `score >= 0.88`
2. Есть хотя бы один риск-сигнал:

- `Exports30m + Denied10m > 0`, или
- `SensitiveViews5m + Mutations5m >= 14`, или
- `Requests1m >= 120`

Таким образом, модель не реагирует на любой высокий score без операционного подтверждения риск-поведения.

## 7. Автомат и ограничения по времени

Состояние пользователя:

- `stepup_required`
- `password_verified`
- `failed_stepups`
- `locked_until`
- `last_triggered_at`
- `last_verified_at`
- `last_risk_score`

Дополнительные guard-условия:

- Не триггерить повторно слишком часто (cooldown после trigger).
- Не оценивать, если уже активирован step-up/lock.
- Не блокировать служебные маршруты (`/api/auth/stepup/*`, logout, ping и пр.).

## 8. Аудит и трассируемость

Ключевые события аудита:

- `security.behavior.stepup_required`
- `security.behavior.stepup_password_ok`
- `security.behavior.stepup_failed`
- `security.behavior.stepup_success`
- `settings.behavior_model.toggle`

Для расследований сохраняются метрики и score, чтобы объяснить причину срабатывания.

## 9. Границы применимости

Модель не заменяет:

- RBAC/ACL,
- контроль сессии,
- 2FA-политику,
- серверные проверки прав.

Она добавляет поведенческий риск-слой поверх существующей безопасности.

## 10. Включение в продукте

Путь в интерфейсе:

`Настройки -> Укрепление безопасности -> Включить поведенческую модель аномалий`

Если флаг выключен, scoring и step-up по поведенческой модели не применяются.
